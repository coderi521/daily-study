## TDD（Test-Driven Development）—— 测试驱动开发

**核心思想**：先写测试，再写实现代码。

### 经典流程（红-绿-重构）

1. **🔴 Red（写一个失败的测试）** — 先写一个测试用例，描述你想要的功能，此时还没有实现代码，测试会失败
2. **🟢 Green（让测试通过）** — 写最简的代码让测试通过，不追求完美
3. **🔵 Refactor（重构）** — 在测试的保护下，安全地重构代码，提高质量

### 优点
- 代码质量更高，测试覆盖率高
- 重构时更有信心（测试就是安全网）
- 倒逼设计更清晰的接口
- 适合需求明确的功能开发

### 缺点
- 学习曲线较陡
- 过度测试可能导致维护成本增加
- 不适合 UI 变化频繁的场景

## SDD（Specification-Driven Development）—— 规范驱动开发

**核心思想**：先定义规范和契约，再实现代码。

SDD 有两种常见的解释：

### 1. 规格说明驱动开发
- 先编写详细的需求规格说明（Specification）
- 经过评审确认后，再进行开发和测试
- 适合大型项目、安全关键系统（如航空、医疗）

### 2. 契约/API 驱动开发
- 先定义 API 接口契约（OpenAPI、Protobuf、gRPC）
- 前后端可以并行开发，彼此解耦
- 接口文档即是"规范"

### 优点
- 减少沟通成本，团队对齐一致
- 适合微服务架构中的多团队协作
- 便于自动化测试和文档生成

### 缺点
- 前期投入大，变更成本高
- 可能陷入"过度设计"
- 不适合快速迭代的实验性项目

---

## 对比总结

| 维度 | TDD（测试驱动） | SDD（规范驱动） |
|------|----------------|----------------|
| **侧重点** | 代码正确性 | 系统一致性 |
| **先做什么** | 先写测试 | 先写规范 |
| **约束对象** | 开发者 | 所有涉众（团队/业务） |
| **变更灵活性** | 高（测试辅助重构） | 低（规范需评审） |
| **适用场景** | 功能开发、库/框架 | 多人团队、接口协作、安全系统 |

---

### 补充：与其他开发方法的关系

- **BDD（Behavior-Driven Development）** — TDD 的演进版，用自然语言描述行为，重在业务价值
- **DDD（Domain-Driven Design）** — 领域驱动设计，关注业务模型而非测试或规范
- **FDD（Feature-Driven Development）** — 功能驱动开发，关注功能列表的迭代交付
