## TL;DR

| 维度   | 传统 D2C           | Tarot Pixel                   |
| ---- | ---------------- | ----------------------------- |
| 核心理念 | 建管道：设计稿 → 代码     | 建图书馆：让 Agent 自己查阅设计稿          |
| 代码生成 | 平台替 Agent 生成代码   | Agent 自己结合项目上下文写代码            |
| 信息传递 | 全量推送 IR 数据       | REST API 按需查询，分层获取            |
| 人工配合 | 大量手动（选图层、切图、标状态） | 只需指出「哪里不对」，Agent 自己修正         |
| 迭代方式 | 一次性输出，不完美重新跑     | 持续在线，可实现→比对→修正闭环              |
| 扩展方式 | 中心化：平台更新规则引擎     | 去中心化：工程层提供新标签，Agent 侧自动利用模型进步 |

## 一、背景：为什么需要新方案？

### D2C 的结构性矛盾

传统 D2C 平台在标准化的 B 端场景尚可控制，但存在**结构性断点**：

| 问题 | 说明 |
|------|------|
| 🧩 图层整理与打标 | 手动打开设计工具选择目标区域，稍有遗漏就缺信息，多选引入噪声 |
| ✂️ 切图与合图 | 哪些该切图、哪些该用 CSS、哪些该合并，需人工判断 |
| 🔄 多状态识别 | 同一组件不同状态是多个独立画板，需人工识别并手动合并 |
| 📦 一次性输出 | 生码后设计稿信息使命结束，还原度不够只能重走流程 |
| 🎨 视觉复杂度高 | C 端一个"红包组件"可能 20 个图层叠加，需逐层判断 |

### 视觉还原不是孤立任务

真正交付一个前端功能，视觉还原只是其中一个环节，必须与以下内容在同一套实现中完成：

- **业务状态耦合**：任务完成、优惠券过期等状态决定视觉表现
- **交互逻辑耦合**：弹窗、Tab 切换、过渡动画
- **数据流耦合**：接口获取价格、库存实时变化

> 单独跑一遍 D2C 平台生成的代码，往往只是个空壳，离「能用」还有很大距离。

## 二、核心理念：Agent-Native 设计

### 从 Vibe Coding 到 Agentic Engineering

| 阶段 | 特征 |
|------|------|
| Vibe Coding | 依赖直觉，和模型一起快速试错 |
| Agentic Engineering | Coding Agent 能 30-60 分钟独立推进完整任务，跨越文件重构、调用工具链、测试调试 |

### 三个关键事实

1. **Coding Agent 已不只是补全代码**，而是能在 30-60 分钟内独立推进完整任务
2. **开发者角色在变**，不是从写代码转架构，而是把判断逻辑转化为 Agent 可执行的约束
3. **Agent 真正价值**在于「观察 → 规划 → 执行 → 反思」的自主循环

### 视觉还原本质是上下文问题

- **Specflow**（同一作者上一篇文章）解决的是「需求理解和任务拆解」上下文
- **Tarot Pixel** 解决的是「视觉信息」上下文

> 模型能力和工具链由平台方迭代，但**上下文质量**是掌握在业务团队手里的核心变量。

### 工具是做给 AI 用的

《The Complete Guide to Agentic Coding 2026》设计原则：

- 工具用自然格式（prose、markdown）而非复杂结构化格式
- 提供充分的文档
- 优雅处理错误
- 假设 Agent 可能以意想不到的方式使用它们

## 三、系统架构

![[附件/excalidraw/2026-07-14-Tarot-Pixel-架构图.svg]]

### 四大核心模块

| 模块 | 功能 |
|------|------|
| **MasterGo 插件** | 数据层核心，解析和降噪设计稿。支持一次性导出整个页面，自动蒙版翻译、矢量形状识别、装饰图层标记、资源导出 |
| **Viewer Server** | 可视化中间层。将设计稿渲染为一比一 Web 界面，支持开发者预览+Agent Browser Use+Map 模式 |
| **API 服务** | 本地 HTTP 服务，20+ REST API。按需拉取、分层获取、幂等无状态 |
| **内置 Agent（Chat API）** | 作为 Coding Agent 的「视觉顾问」，处理图层定位、切图决策、还原度比对 |

### API 分层设计

```
第1层 — overview / screenshot        → 整体结构概览
第2层 — d2c-context                  → 选定节点的结构化数据
第3层 — composite / node-map         → 合成图、图片定位
第4层 — chat                         → 视觉语义分析、比对
```

> **关键原则**：实现一个简单按钮只需要第 1/2 层，复杂卡片才到第 3/4 层。按需获取，上下文开销极低。

## 四、Skill 驱动协作

Tarot Pixel 通过 **SKILL.md** 文档赋予 Coding Agent 新能力：

```
/skills/tarot-pixel/SKILL.md
└─ 纯 API 参考文档
   ├─ 告知有哪些 API（overview, d2c-context, composite, screenshot, chat, node-map）
   ├─ 每个 API 返回什么、什么时候用
   ├─ 通用原则（装饰优先切图、内容优先代码、所有数值必须来自 API）
   └─ 标准工作流参考（定位 → 取证 → 决策）
```

**不绑定**：不规定代码风格、不规定用什么组件库、不定义项目实现规范。

### 中心化 vs 去中心化扩展

| 方式 | 传统 D2C | Tarot Pixel (Skill 模式) |
|------|----------|------------------------|
| 扩展方式 | 更新规则引擎、代码模板 | 工程层提供新数据标签/API |
| 智能集中 | 平台侧 | Agent 侧 |
| 维护成本 | 随场景线性增长 | 模型进步自动提升效果 |
| 优势 | 可控性强 | 模型变聪明 → 系统自动变好 |

## 五、修正闭环

### 两种修正方式

**开发者提示修正**
```
开发者 → "这里的间距不对"
Coding Agent → 查设计稿 → 修正 → 再比对
```

**自主修正**
```
Coding Agent
  → 调 screenshot API 与设计稿对比
  → 调 d2c-context 查看遗漏细节
  → 调 composite 获取合成图修正背景
  → 通过 node-map 用图片识别定位需修正节点
```

### 真正的效率指标：人工干预次数

> 从需求到交付，开发者需要介入多少次。
> - **Specflow**：减少需求拆解和任务编排的干预
> - **Tarot Pixel**：减少视觉还原和修正环节的干预

## 六、与 MasterGo MCP 的差异

| 维度 | MasterGo MCP | Tarot Pixel |
|------|-------------|-------------|
| 预处理 | 需手动选容器、处理分组 | 一次导出整个页面，自动处理 |
| 数据返回 | 全量 JSON 注入上下文 | 分层 API，按需查询 |
| 辅助能力 | 仅数据获取 | 20+ API（截图、合图、定位、比对） |
| 生命周期 | 一次性获取 | 持续在线，随时回查 |

## 七、当前进展与计划

- ✅ Viewer Server：20+ REST API
- ✅ Node-Map：图片识别定位节点

- ❌ 多状态识别：准确率持续优化中

### 未来计划
- 🔜 更智能的降噪
- 🔜 更快速的图层定位
- 🔜 等待模型进化带来架构前瞻性红利

## 八、总结

> **D2C 的核心问题不是不能生成代码，而是自动化链路中仍有大量环节依赖人工配合。**

| 核心原则 | 说明 |
|---------|------|
| 不替 Agent 做决策 | 代码生成完全交给 Coding Agent |
| 按需查询，而非全量推送 | 多层 API 设计，上下文开销低 |
| 工程确定性绝不交给 AI | 像素级精确的 CSS 提取靠算法 |
| AI 只处理理解力和判断力 | 装饰/内容识别、状态关系、还原度比对 |
| 持续在线不追求一次性完美 | 设计稿始终可查，闭环迭代 |
