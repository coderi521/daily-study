# MCP 与 Function Calling 的区别

## 一句话概括

> **Function Calling** 是 LLM API 的一个**功能**，让模型能输出结构化函数调用请求；**MCP（Model Context Protocol）** 是一个**开放协议**，定义了 AI 应用与外部工具/数据源交互的完整标准。

---

## 核心对比

| 维度 | Function Calling | MCP |
|------|------------------|-----|
| **本质** | API 功能特性 | 开放标准协议 |
| **提出者** | OpenAI（后被各大厂商跟进） | Anthropic |
| **层级** | API 层的工具调用能力 | 应用层的通信协议 |
| **范围** | 单一：定义工具 → 模型输出调用 | 全链路：工具发现、调用、资源管理、传输层 |
| **标准化程度** | 各厂商实现不同（格式、参数各异） | 统一标准，跨平台跨模型 |
| **传输方式** | HTTP 请求-响应（无状态） | 支持 stdio、SSE 等多种传输（有状态长连接） |
| **连接方式** | 每次请求传入工具定义 | 可建立持久连接，复用上下文 |
| **生态** | 单点集成，需自己封装 | 完整生态：Server、Client、Transport 分层架构 |
| **扩展性** | 需手动管理工具注册和发现 | 内置服务发现、能力协商、资源管理 |

---

## 详细解析

### 1. 概念层级不同

**Function Calling** 是模型 API 的一个 **输出格式控制能力**。当你在 API 请求中传入工具定义（tools 参数），模型会在需要时返回一个 `tool_calls` 结构化对象，而不是纯文本回复。它解决的是"让模型输出结构化指令"这个问题。

```python
# Function Calling：在 API 请求中传入工具定义
response = client.messages.create(
    model="claude-sonnet-4-20250514",
    tools=[{
        "name": "get_weather",
        "description": "获取指定城市的天气",
        "input_schema": {
            "type": "object",
            "properties": {
                "city": {"type": "string"}
            }
        }
    }]
)
# 模型返回 tool_use 内容块
```

**MCP** 是一个完整的 **协议层标准**，它不仅包含工具调用，还定义了：
- **工具（Tools）**：可供模型调用的功能单元
- **资源（Resources）**：可供模型读取的数据源（文件、数据库等）
- **提示（Prompts）**：预定义的提示模板
- **传输层（Transports）**：通信方式（stdio、SSE、Streamable HTTP）
- **生命周期管理**：服务发现、初始化、能力协商、关闭

### 2. 集成方式不同

| 特性 | Function Calling | MCP |
|------|------------------|-----|
| 工具定义位置 | 每次请求传入 API | 在 MCP Server 端声明 |
| 工具发现 | 开发者手动编写 tools 参数 | 自动通过 `tools/list` 端点发现 |
| 调用流程 | 客户端完成调用后传给模型 | MCP Host → Server 标准化交互 |
| 多工具管理 | 需自己维护工具列表 | Server 统一管理，可动态增删 |

### 3. 工作模式差异

```
Function Calling 模式（无状态）：
┌─────────┐    请求+tools    ┌──────────┐
│ 客户端   │ ──────────────→ │  LLM API  │
│         │ ←────────────── │          │
│         │  返回 tool_use   │          │
│         │                 └──────────┘
│         │    执行工具调用
│         │    再次请求+结果
│         │        ...
```

```
MCP 模式（有状态长连接）：
┌──────────┐                  ┌──────────┐
│ MCP Host │ ── 初始化连接 ─→ │ MCP Server│
│ (客户端)  │ ←── 能力声明 ─── │          │
│          │                  │          │
│          │ ── tools/list ─→ │   工具    │
│          │ ←── 工具列表 ─── │   资源    │
│          │ ── tools/call ─→│   提示    │
│          │ ←── 调用结果 ─── │          │
│          │      ...        │          │
└──────────┘                  └──────────┘
       ↕ LLM 通过 MCP Host 与 Server 交互
```

### 4. 适用场景

**Function Calling 更适合**：
- 简单的工具调用（如查询天气、搜索）
- 快速的 API 集成
- 不需要复杂上下文管理的场景

**MCP 更适合**：
- 需要访问多种数据源的复杂应用（文件系统、数据库、API）
- 需要持久连接和实时更新的场景
- 跨平台、跨模型的工具生态构建
- 企业级工具集成平台

---

## 一句话总结

|     | Function Calling                        | MCP                |
| --- | --------------------------------------- | ------------------ |
| 好比  | **一个函数**                                | **整个微服务框架**        |
| 功能  | 让模型能调用工具                                | 定义工具怎么被发现、连接、调用、管理 |
| 关系  | MCP 的传输层可以用 Function Calling 来实现，两者可以协作 |                    |

> **简单理解**：Function Calling 解决"模型如何调用工具"的问题，MCP 解决"AI 应用如何与外部世界标准化连接"的问题。MCP 是 Function Calling 的超集和标准化升级。
